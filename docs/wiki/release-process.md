# 🚀 Release Process

This document defines the **official release process** for `awsctl`. The goal of this process is to ensure that every release is predictable, auditable, reversible, and safe to deploy in regulated environments.

This document is authoritative.

---

## 🏗️ Release Philosophy

`awsctl` is a **security-sensitive client**. Releases prioritize correctness and safety over speed.

* **Correctness:** Functional behavior must match specifications.
* **Safety:** No regressions in the trust or security models.
* **Backward Compatibility:** Existing workflows should not break unexpectedly.
* **Auditability:** Every change must be traceable and justifiable.

---

## 🔢 Versioning Scheme

`awsctl` follows **Semantic Versioning (SemVer)**: `MAJOR.MINOR.PATCH`.



* **MAJOR:** Breaking behavior or fundamental trust model changes.
* **MINOR:** New features, new guardrails, or significant capability updates.
* **PATCH:** Bug fixes, documentation updates, or internal refactors.

Any change affecting **trust boundaries, credential handling, or guardrails** requires at least a **MINOR** version bump.

---

## 📦 Release Artifacts

No release is considered valid without the following artifacts:

* **Tagged Git Commit:** Immutable point in history.
* **Signed GitHub Release:** Cryptographic proof of origin.
* **Updated CHANGELOG:** Clear record of changes.
* **Versioned Binaries:** Compiled artifacts for supported architectures.
* **Verified Install Scripts:** Updated `brew` formulae or installation scripts.

---

## 🔄 Release Workflow

The workflow follows a strict path from preparation to validation.



### Step 1: Prepare Release
* Update the version in the source code.
* Consolidate the `CHANGELOG`.
* Run the full test suite and coverage gates.

### Step 2: Tag Release
```bash
git tag -a vX.Y.Z -m "Release vX.Y.Z"
git push origin vX.Y.Z
```
> **Note:** Tags are immutable once pushed to the remote repository.

### Step 3: Publish Release
* Create a GitHub release from the tag.
* Attach compiled binaries and **SHA-256 checksums**.
* Include the finalized release notes.

### Step 4: Validate Installation
Validation must occur on a clean machine or container:

```bash
brew install awsctl
awsctl version
awsctl doctor
```

---

## 🛡️ Stability and Safety Gates

### Hotfix Process
Hotfixes land on `main` and are strictly reserved for:
1.  **Security vulnerabilities.**
2.  **Data corruption risks.**
3.  **Critical usability regressions.**

### Rollback Strategy
`awsctl` is inherently easy to roll back because it is a client-side tool with no server-side state.

1.  **Uninstall current version:** `brew uninstall awsctl`
2.  **Install previous version:** `brew install awsctl@<previous_version>`

---

## ✅ Release Invariants (Non-Negotiable)

The following actions are strictly forbidden and invalidate a release:

* **Force-pushing release tags.**
* **Silent behavior changes** (undocumented features).
* **Skipping security tests** or coverage gates.
* **Out-of-band binaries** (manual uploads not generated by CI).

---

## 🧾 Audit Evidence

Every release generates a trail of evidence for auditors:

* **Git tag history:** Who tagged the release and when.
* **CI Logs:** Proof that all tests passed for the specific commit.
* **GitHub Metadata:** Timestamped release information.

> [!IMPORTANT]
> If releasing `awsctl` ever feels “fast and loose,” the process has been violated. This document defines the standard required to maintain the **Root of Trust**.
